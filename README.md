# GPS/GNSS Drone Monitoring Prototype

Демонстрационный прототип для практической части диплома:
разработка системы мониторинга и визуализации перемещений дронов доставки на основе GPS/GNSS-данных.

Проект не переписан с нуля. За основу взят существующий репозиторий маршрутизации и live-симуляции дронов на `FastAPI`, поверх которого добавлен отдельный функциональный слой мониторинга миссий, обработки телеметрии и диспетчерского интерфейса.

## Что уже есть в проекте

- `FastAPI` backend и `WebSocket`-обновления в реальном времени.
- `Leaflet`-интерфейс с картой и текущим базовым UI.
- Построение маршрутов по графу города.
- Поддержка нескольких дронов.
- Учет батареи и видимые станции зарядки.
- Обход запретных зон.

## Что добавлено для дипломной части

- отдельный слой `drone_monitoring/` с persistent-хранилищем на `SQLite`;
- сущности `missions`, `drones`, `orders`, `track_points_raw`, `track_points_processed`, `processing_runs`, `mission_events`, `mission_metrics`, `geozones`, `users`;
- synthetic mission generation для Волгограда;
- live-миссии с записью поступающих точек телеметрии;
- асинхронная обработка тяжёлых запросов интерфейса и фоновая очередь записи live-телеметрии в БД;
- предобработка GPS/GNSS-трека, интерполяция пропусков, outlier filtering и базовый Kalman;
- расчёт метрик `RMSE`, `CEP50`, доли пропусков, числа аномалий и среднего `HDOP`;
- диспетчерская страница `/monitor`;
- страница постановки миссии по карте `/mission-setup`;
- изменение точки доставки во время полёта;
- несколько запретных зон;
- немедленное перепрокладывание маршрута при появлении новой no-fly зоны;
- разделение списка миссий на активные и архив;
- отмена миссии с сохранением причины в истории;
- короткая сервисная пауза дрона в точке старта и в точке выдачи для имитации забора/выдачи груза;
- аварийная посадка при полной потере заряда;
- сохранение диспетчерских действий и событий миссии в БД.

## Технологический стек

- Python `3.11.9`
- Poetry
- Docker / Docker Compose
- FastAPI
- Leaflet
- SQLite
- логическая совместимость модели данных с PostgreSQL/PostGIS

## Структура проекта

- `api_server.py` — основной FastAPI runtime, симуляция, WebSocket, старый API и точки интеграции.
- `data_service.py` — загрузка и подготовка данных города.
- `graph_service.py` — граф города и обработка запретных зон.
- `routing_service.py` — кратчайшие безопасные маршруты и логика зарядки.
- `drone_monitoring/db.py` — инициализация SQLite и схема таблиц.
- `drone_monitoring/service.py` — миссии, треки, события, метрики, geozones, live hooks.
- `drone_monitoring/processing.py` — фильтрация, интерполяция и базовая обработка треков.
- `drone_monitoring/api.py` — новые API-маршруты дипломной части.
- `static/mission_monitor.html` — диспетчерский интерфейс.
- `static/mission_setup.html` — постановка миссии по карте.
- `static/index.html` — исходный UI репозитория, сохранён и расширен ссылками на новые страницы.

## Требования перед запуском

### Для локального запуска

- установлен Python `3.11.9`;
- установлен Poetry;
- свободен порт `8000`.

### Для запуска через Docker

- установлен Docker Desktop;
- Docker Desktop запущен;
- свободны порты `8000` и `6379`.

## Запуск локально через Poetry

### Windows PowerShell

```powershell
poetry env use 3.11
poetry install
poetry run uvicorn api_server:app --host 0.0.0.0 --port 8000 --reload
```

### Что открыть после запуска

- базовый интерфейс: `http://localhost:8000/`
- диспетчерская: `http://localhost:8000/monitor`
- постановка миссии: `http://localhost:8000/mission-setup`

## Запуск через Docker Compose одной командой

Это основной рекомендованный способ демонстрации проекта.

### Команда запуска

```powershell
docker compose up --build
```

### Что произойдёт

1. Docker соберёт образ приложения.
2. Поднимется контейнер `redis`.
3. Поднимется контейнер `app` с FastAPI в одном worker-процессе.
4. SQLite-база будет храниться в папке `./data`.

Один worker используется намеренно: live-симуляция хранит текущее состояние дронов в общей памяти процесса, поэтому многопроцессный запуск приводил бы к рассинхронизации карты и миссий.

Дополнительно внутри приложения уже используется неблокирующая схема работы:

- тяжёлые операции чтения миссий и треков вынесены в `asyncio.to_thread(...)`;
- запись live-телеметрии идёт через отдельную очередь;
- SQLite работает в режиме `WAL`, что уменьшает блокировки между UI и фоновыми задачами.

### Как понять, что всё запустилось

В логах должен появиться запуск `Uvicorn`, после чего можно открыть:

- `http://localhost:8000/`
- `http://localhost:8000/monitor`
- `http://localhost:8000/mission-setup`

### Как остановить

Если запуск был в текущем окне терминала:

- нажмите `Ctrl + C`

Затем для корректной остановки контейнеров:

```powershell
docker compose down
```

### Как запустить повторно без пересборки

```powershell
docker compose up
```

### Как полностью сбросить состояние

Если нужно начать демонстрацию “с чистого листа”:

```powershell
docker compose down -v
```

После этого при необходимости можно удалить локальную папку `data`, если хотите сбросить историю миссий и SQLite-базу.

Если вы уже запускали более старую версию проекта и видите “зависшие” старые миссии, сначала выполните именно этот полный сброс, а затем снова:

```powershell
docker compose up --build
```

### Где лежат данные

- `./data` — SQLite-база дипломной части;
- `./cache_data` и `./cache` — кэш исходного репозитория;
- Redis используется как runtime persistence для существующего функционала.

### Если запуск не удался

Проверьте:

- что Docker Desktop действительно запущен;
- что порты `8000` и `6379` не заняты другими программами;
- что вы запускаете команду из корня проекта, где лежит `docker-compose.yml`.

## Диспетчерский сценарий демонстрации

Рекомендуемый сценарий показа на защите:

1. Открыть `/monitor`.
2. Показать, что карта работает в Волгограде и видны станции зарядки.
3. Перейти на `/mission-setup`.
4. Задать точку старта, точку доставки и несколько запретных зон.
5. Запустить live-миссию.
6. Вернуться на `/monitor` и показать:
   - список активных миссий;
   - список дронов;
   - live-положение дрона на карте;
   - raw и processed track;
   - delivery zone и no-fly zones;
   - события миссии;
   - метрики;
   - карточку дрона и логи.
7. Во время полёта изменить точку доставки и показать немедленное перепрокладывание маршрута.
8. Добавить новую no-fly зону поверх текущего маршрута и показать уход дрона из опасной области.
9. Отменить миссию и показать, что причина отмены сохраняется в истории, а её траектории и точки убираются с карты.
10. Показать, что при критическом разряде дрон уходит на аварийную посадку, а событие сохраняется в журнале.

## Текущее поведение маршрутизации

- доставка строится по кратчайшему безопасному воздушному маршруту из доступных;
- несколько no-fly зон поддерживаются одновременно;
- точки старта и доставки, выбранные на карте, сохраняются по точным координатам пользователя;
- если новая запретная зона появляется на текущей траектории, активный дрон перепрокладывает маршрут;
- если дрон уже находится внутри новой запретной зоны, он выходит из неё кратчайшим безопасным путём;
- при прибытии в точку забора и в точку выдачи дрон делает короткую остановку примерно на 5 секунд;
- заказ считается завершённым в момент выдачи, а дальнейший перелёт дрона к зарядке является уже пост-миссионным этапом;
- если заряд снижается до `20%` и ниже, сохраняется логика перелёта к станции зарядки;
- после зарядки активная миссия перепланируется заново от текущего положения дрона, а не по старому пути;
- если дрон полностью теряет заряд до достижения станции, формируется аварийное завершение и событие `EMERGENCY_LANDING`.

## Что видит диспетчер

- по умолчанию в `/monitor` отображаются только активные миссии;
- завершённые, отменённые и аварийно завершённые миссии доступны через кнопку архива;
- при выборе миссии на карте показываются её старт, точка выдачи, плановая и фактическая траектория;
- зарядные станции, дроны и актуальные запретные зоны остаются видимыми на карте постоянно;
- миссию можно отменить с указанием причины, и эта причина попадёт в журнал событий.

## Примечание по воспроизводимости

Проект рассчитан как демонстрационный, но рабочий и воспроизводимый:

- новые функции вынесены в отдельный модуль, а не размазаны по проекту;
- старый функционал маршрутизации сохранён;
- новая БД использует SQLite, но модель спроектирована так, чтобы её можно было переносить в PostgreSQL/PostGIS;
- город визуализации по умолчанию — Волгоград.
